# Инструкции для Claude Code

## Git-дисциплина

**Рабочая ветка:** вся разработка ведётся только в рабочей ветке. Никогда не коммитить напрямую в `main`.

- Один коммит = одно логически завершённое изменение. Не накапливать несколько правок в одном коммите.
- После каждого коммита — сразу `git push`. Код без пуша не существует для CI, деплоя и остальных.
- Формула: `git add <files> && git commit -m "..." && git push origin <branch>`
- Если сессия длится несколько часов — коммитить и пушить минимум раз в день, даже незаконченное (WIP-коммит).

## Документация

Обновлять документы сразу по факту, не копить в конце сессии:

- **ARCHITECTURE.md** — при любом изменении флоу, структуры данных, API или ключевых зависимостей
- **CHANGELOG.md** — после каждой новой фичи или изменения поведения, пока контекст свежий
- **README.md** — при изменении стека, публичных эндпоинтов или способа развёртывания
- **BACKLOG.md** — при закрытии задачи, появлении нового долга или изменении приоритетов

Незакоммиченные изменения и необновлённая документация в конце сессии — незавершённая задача.

## Деплой

- Деплоить сразу после изменения кода — не заканчивать ответ фразой "нужно задеплоить"
- Код без деплоя = ничего не изменилось с точки зрения пользователя
- Знать платформо-специфичные флаги и ограничения инструментов деплоя (MCP vs CLI)

## Перед реализацией — сначала понять текущее состояние

Перед любым изменением инфраструктуры, интеграций или существующего поведения:

1. Понять как система работает **прямо сейчас**, не предполагать
2. Спросить или проверить: как именно пользователь сейчас подключается/использует систему?
3. Если что-то работало — спросить "как именно это было настроено?" до того как менять
4. Спросить "тебе это реально нужно?" перед реализацией сложных фич

Три итерации попыток починить то, что работало — признак того что не разобрались в текущем состоянии.

## Безопасность

- Никогда не писать `DELETE` / `UPDATE` без `WHERE` — даже в миграциях
- `DROP COLUMN`, `RENAME COLUMN` — только в два шага: убрать из кода → деплой → менять схему
- При мутациях (изменение, удаление) — всегда проверять что пользователь является владельцем
- Если используется сервисный ключ (bypass RLS) — вся проверка доступа только через код, не через политики БД
- Не обходить функции проверки доступа прямыми запросами

## Стиль кода

- Не добавлять фичи, рефакторинг или абстракции за пределами задачи
- Не добавлять обработку ошибок для сценариев которые не могут произойти
- Комментарии только когда WHY неочевиден — скрытое ограничение, воркэраунд для конкретного бага
- Не объяснять ЧТО делает код — хорошие имена это уже делают
